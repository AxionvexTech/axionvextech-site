# 1. Purpose of This Document

This document defines the complete rebuild of the Axionvex Tech website. It replaces the current layout, page hierarchy, visual style, and most current marketing copy.

The rebuilt website must position Axionvex Tech as a **production AI workflow engineering partner for B2B SaaS and operations-heavy businesses**. The website must show that the agency does more than create AI demonstrations. It designs, builds, deploys, monitors, and improves AI-enabled systems that operate inside real business workflows.

This document is intended to be uploaded directly into Cursor or Codex and used as the primary implementation brief. The coding agent must treat it as the source of truth unless a requirement conflicts with the existing repository, deployment environment, legal obligations, or verified business facts.

## Core rebuild objectives

1. Replace broad “software agency” messaging with a focused production AI workflow position.
2. Create a stronger visual identity with more meaningful imagery, product-interface visuals, system diagrams, and real human context.
3. Increase buyer trust through clear methods, controls, proof standards, and transparent delivery expectations.
4. Separate client acquisition from recruiting so each audience has a clear path.
5. Make the recruiting area configurable without requiring layout or code changes for every role.
6. Improve search visibility through dedicated solution, use-case, case-study, standards, and insight pages.
7. Build a fast, accessible, responsive, maintainable website that can be updated through typed content files or a future CMS.
8. Avoid unsupported claims, fabricated logos, fictional clients, misleading dashboards, fake metrics, and unverified testimonials.

# 3. Brand and Trust Guardrails

These requirements are mandatory across the entire website.

## 3.1 No fabricated proof

Do not publish:

- Fake customer logos
- Fictional case studies presented as real work
- Invented client quotes
- Unverified revenue, uptime, release, conversion, cycle-time, or cost-savings figures
- A fake “live operations console”
- Team headshots that imply real employees when they are generated or stock imagery
- Certifications, partnerships, or compliance claims that the company does not hold

## 3.2 Proof labeling

Every project shown must use one of the following labels:

- **Verified client work**
- **Client name withheld under NDA**
- **Internal system**
- **Technical demonstration**
- **Reference architecture**

A technical demonstration must never be presented as a client engagement.

## 3.3 Claims format

When a measurable result is published, include:

- Baseline condition
- Measurement period
- What Axionvex Tech controlled
- Whether the result is measured, estimated, or client-reported
- Relevant limitations

Example:

> Reduced average manual review time from approximately 18 minutes to 11 minutes during a four-week production pilot. Measured across 1,240 reviewed records. Final performance varied by document quality and exception rate.

## 3.4 Account and identity safety

The public website must not promote account renting, credential sharing, third-party account operation, identity lending, impersonation, location masking, or remote control of a personal account as a service or employment opportunity.

The careers and applicant safety content should state that Axionvex Tech does not require applicants to:

- Share personal platform credentials
- Rent or sell accounts
- Impersonate another person
- Allow hidden access to a personal device
- Pay an application fee
- Purchase equipment from a specified individual

## 3.5 Legal business identity

Before launch, add verified values for:

- Legal company name
- Business jurisdiction
- Registered business address or lawful contact address
- Contact email
- Privacy contact
- Applicant privacy contact
- Terms effective date
- Privacy policy effective date

Use clear placeholders until verified. Do not invent these details.

# 5. Visual Direction

## 5.1 Overall visual concept

**Editorial systems engineering.**

The site should combine:

- Clean editorial typography
- Structured technical diagrams
- Product UI mockups or sanitized real screenshots
- Realistic workplace photography
- Subtle depth and motion
- Strong spacing
- Dark technical surfaces balanced with warm light backgrounds

The site must look premium without appearing like a cryptocurrency project, gaming brand, or futuristic concept site.

## 5.2 Recommended color system

Use CSS variables and support light and dark sections.

| Token | Suggested value | Purpose |
|---|---:|---|
| `--ink-950` | `#07111F` | Main dark background |
| `--ink-900` | `#0D1A2A` | Secondary dark surface |
| `--ink-800` | `#16263A` | Elevated dark cards |
| `--paper-50` | `#F7F9FC` | Main light background |
| `--paper-100` | `#EEF2F6` | Light card background |
| `--white` | `#FFFFFF` | High contrast text/surfaces |
| `--slate-600` | `#5C6B7A` | Secondary light text |
| `--slate-300` | `#B7C2CE` | Secondary dark text |
| `--signal-mint` | `#6EE7C8` | Primary accent and success |
| `--signal-blue` | `#7C9CFF` | Secondary accent |
| `--signal-amber` | `#F2C66D` | Warning, highlighted data |
| `--danger` | `#EF7A7A` | Error and risk state |

Do not use large uncontrolled gradients. Small gradients may be used inside diagrams, glows, or background atmosphere at low opacity.

## 5.3 Typography

Use one modern sans-serif family with strong variable weights and one optional monospace family.

Recommended approach:

- Primary: Geist, Inter, Manrope, or another high-quality variable sans-serif
- Monospace: Geist Mono, IBM Plex Mono, or JetBrains Mono

Typography rules:

- Hero heading should be large but not oversized to the point of breaking comprehension.
- Headings should use tight line height and restrained letter spacing.
- Body text should remain 16 to 19 pixels on desktop, at least 16 pixels on mobile.
- Avoid all-uppercase paragraphs.
- Use monospace only for labels, architecture nodes, metrics, tags, and technical annotations.

## 5.4 Grid and spacing

- Maximum content width: 1200 to 1280 pixels
- Main text measure: 620 to 760 pixels
- Standard section padding: 96 to 128 pixels desktop, 64 to 80 pixels tablet, 48 to 64 pixels mobile
- Use a 12-column desktop grid
- Use clear whitespace between content groups
- Avoid dense card walls

## 5.5 Corners and borders

- Moderate radius: 14 to 22 pixels
- Use borders and surface changes before heavy shadows
- Shadows should be soft and restrained
- Technical diagrams may use square or slightly rounded nodes

## 5.6 Motion

Motion must clarify hierarchy, not decorate the page.

Use:

- Small fade and translate entrance effects
- Line drawing or node activation inside the hero workflow diagram
- Hover elevation on cards
- Subtle number or status transitions
- Respect `prefers-reduced-motion`

Do not use:

- Excessive parallax
- Continuous spinning objects
- Cursor-following effects
- Page transitions that delay navigation
- Large video backgrounds

> Create a premium editorial product-interface visual for a production AI engineering agency. On the output side, show a CRM update, a generated operations report, a routed support case, and an audit log. Include a restrained observability bar for evaluation quality, exceptions, latency, and cost. Dark graphite and deep navy interface, warm off-white labels, mint and muted blue accents. Clean, credible, enterprise software aesthetic. No robots, no glowing brain, no sci-fi holograms, no cryptocurrency style, no unreadable tiny text. Wide 16:10 composition.

# 21. AI Standards Page

## Hero

**Eyebrow**  
AI Standards

**Heading**  
# Production AI should be observable, controlled, and owned.

**Body**  
These standards guide how Axionvex Tech designs, evaluates, deploys, and operates AI-enabled workflows. The exact controls vary by business consequence, data sensitivity, and user need.

## 21.1 Workflow suitability

Before implementation, determine:

- Is the work repetitive enough to justify systemization?
- Are inputs available and lawful to use?
- Can quality be evaluated?
- Is the expected outcome measurable?
- Are exception paths known?
- Can a person review uncertain or high-impact decisions?
- Would deterministic automation be more appropriate?

## 21.2 Model strategy

- Avoid unnecessary vendor lock-in
- Use task-appropriate models
- Route by quality, latency, cost, and data constraints
- Keep model configuration versioned
- Define fallback behavior
- Test changes before broad release

## 21.3 Context and retrieval

- Use approved sources
- Track source identity and freshness
- Apply role and tenant boundaries
- Limit context to what the task requires
- Cite sources where the user needs verification
- Define behavior when evidence is missing

## 21.4 Tool access

- Least privilege
- Explicit action allowlists
- Validated inputs and outputs
- Idempotency where applicable
- Rate-limit handling
- Secrets management
- Audit logging
- Human confirmation for high-impact actions

## 21.5 Human approval

Use human approval based on consequence, uncertainty, reversibility, and policy.

Examples that may require approval:

- Financial action
- Customer commitment
- Legal or compliance judgment
- Sensitive communication
- Account or permission change
- Irreversible system update
- Low-confidence exception

## 21.6 Evaluation

Every production workflow should define:

- Expected behavior
- Failure taxonomy
- Representative cases
- Edge cases
- Adversarial or misuse cases where relevant
- Release thresholds
- Regression schedule
- Human review method

Evaluation dimensions may include:

- Correctness
- Completeness
- Grounding
- Policy compliance
- Tool selection
- Action accuracy
- Escalation quality
- Tone
- Latency
- Cost

## 21.7 Observability

Capture enough information to answer:

- What input triggered the workflow?
- What context was used?
- What model and configuration ran?
- What tools were called?
- What rule or person approved the action?
- What failed or required retry?
- What did the workflow cost?
- What was the final outcome?

## 21.8 Security and privacy

- Data minimization
- Encryption in transit and at rest
- Role and tenant isolation
- Retention rules
- Secrets management
- Vendor data-use review
- Incident logging
- Access review
- Environment separation

Do not make compliance certifications unless verified.

## 21.9 Cost and latency

- Establish cost per completed workflow
- Track cost by model and stage
- Use caching where appropriate
- Route simple tasks to appropriate models
- Set timeouts and fallbacks
- Avoid repeated context transfer
- Review expensive failure loops

## 21.10 Ownership

Every system should have:

- Technical owner
- Business owner
- Runbook
- Architecture record
- Integration inventory
- Model and prompt version history
- Incident process
- Change approval process
- Support agreement

# 29. Technical Implementation Direction

Use the existing repository and deployment environment as the starting point. The coding agent must inspect:

- Deployment configuration
- Analytics
- Form handling
- Existing content and assets

## Preferred stack when a full replacement is justified

- Next.js App Router
- TypeScript with strict mode

- Server components by default
- Client components only for interactivity
- MDX or typed content collections for case studies and insights
- Schema validation for forms and content
- First-party or privacy-respecting analytics
- Deployment compatible with the current host

Do not force exact package versions from this document. Use stable versions compatible with the repository and verify official documentation during implementation.

- Avoid large animation libraries unless needed
- Avoid a component library that imposes a generic visual identity
- Use accessible primitives where helpful
- Remove unused dependencies after migration
- Do not add a CMS unless the content-maintenance need justifies it

## Rendering strategy

- Static generation for marketing pages where possible
- Incremental or scheduled regeneration for content updates if supported
- Server-render job and article detail pages
- Keep client-side JavaScript low

## Forms

- Validate on client and server
- Sanitize and normalize input
- Protect against spam and abuse
- Handle success and failure states accessibly
- Log operational errors without storing unnecessary sensitive content

## Image handling

- Use framework image optimization where available
- Preserve aspect ratio
- Avoid layout shift
- Provide meaningful alt text
- Track image source and license

# 31. Accessibility Requirements

Target WCAG 2.2 AA practices.

Mandatory:

- Semantic landmarks
- Logical heading order
- Keyboard-accessible navigation and dialogs
- Visible focus indicators
- Sufficient contrast
- Reduced-motion support
- Descriptive link text
- Form labels and error associations
- Accessible status messages
- Alt text strategy
- No information communicated by color alone
- Skip-to-content link
- Accessible mobile menu
- Captions or transcripts for video
- SVG diagrams with text alternatives or adjacent descriptions

Automated testing does not replace manual keyboard and screen-reader review.

# 32. Performance Requirements

## Targets

Aim for strong Core Web Vitals under real mobile conditions.

- Avoid heavy JavaScript on marketing pages
- Keep LCP content server-rendered
- Optimize hero media
- Prevent cumulative layout shift
- Lazy-load below-fold assets
- Self-host fonts where lawful and practical
- Use font subsetting
- Preload only critical resources
- Avoid third-party script bloat

## Performance budget

Set and enforce a project-specific budget. Recommended starting point:

- Minimal client JavaScript on static pages
- Hero media optimized for the target viewport
- No autoplay background video
- No large uncompressed PNG screenshots
- No duplicate font families

Run Lighthouse and framework build analysis before launch.
